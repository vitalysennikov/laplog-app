# Задача: обновление приложения требует удаления (отложено)

## Проблема

Пользователь не может обновить приложение поверх установленного — Android
требует удалить старую версию перед установкой новой. Это должно означать
конфликт подписи APK (`INSTALL_FAILED_UPDATE_INCOMPATIBLE` или похожее).

## Что уже проверено и НЕ является причиной

- `.github/debug.keystore` закоммичен один раз, коммит `d3b7bbb`
  (2026-05-29, «CI: стабильный debug keystore для консистентной подписи APK»),
  и с тех пор ни разу не менялся (проверено сравнением sha256 файла по всей
  истории git — хэш один и тот же везде).
- До этого коммита GitHub Actions генерировал случайный debug-ключ на каждом
  запуске (раннеры одноразовые), поэтому каждая сборка до 29 мая была
  подписана иначе, чем предыдущая — ожидаемо требовала удаления.
- Гипотеза «текущий установленный APK старше 29 мая, один раз переустановить
  и дальше будет ОК» — **проверена пользователем и не подтвердилась**:
  проблема сохраняется даже после стабилизации ключа, т.е. дело не только в
  разовом переходе на новый ключ.

## Гипотезы, которые ещё не проверялись

- `versionCode`/`versionName` в `app/build.gradle.kts` — не бампается ли он
  непоследовательно (одинаковый/ниже между сборками)? Хотя это обычно даёт
  другую ошибку, не требующую удаления, а просто отказ установки.
- Реальный текст ошибки от Android при попытке обновления — не собран.
  Нужно взять с устройства (через `adb install app-debug.apk` в терминале,
  а не через файловый менеджер/браузер — тогда Android покажет причину в
  выводе adb, например `INSTALL_FAILED_UPDATE_INCOMPATIBLE` или
  `INSTALL_FAILED_VERSION_DOWNGRADE`).
- Возможно, скачивается/устанавливается не тот APK (не с последнего запуска
  GitHub Actions) — стоит явно сверить, что оба APK (старый установленный и
  новый) собраны уже после `d3b7bbb` и подписаны одним и тем же сертификатом:
  ```
  apksigner verify --print-certs old.apk
  apksigner verify --print-certs new.apk
  ```
  (или `keytool -printcert -jarfile app.apk`) — сравнить отпечатки.
- Проверить `applicationId`/`namespace` — не установлено ли на устройстве
  параллельно что-то ещё с тем же package id (например, если когда-то ставился
  release-сборка или сборка из Android Studio с другим debug-keystore
  разработчика, отличным от `.github/debug.keystore`).

## Статус

Отложено пользователем — не блокирует текущую работу над бэкапами/акцентами.
Практическое следствие на время отладки: восстановление из бэкапа после
вынужденного удаления должно быть максимально надёжным (см.
[task_sqlite_raw_backup.md] — там сохранение старого JSON-пути restore
как есть, без изменений, ради этого сценария).
